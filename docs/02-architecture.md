## 1. Що це за продукт (однією фразою)

Telegram-бот, який голосом грає роль власника нерухомості, веде з менеджером
тренувальний діалог (заперечення / сумніви / відмови), записує розмову і після неї
видає звіт з оцінкою навичок.

Замовник у ТЗ **прямо просить документацію**, не лише бота. Тому результат = ДВА:

1. **Документ-пропозиція для власника** (рос/укр, простою мовою) під його 6 пунктів:

## 3. Два питання до власника — ✅ ПІДТВЕРДЖЕНО

1. **Формат голосу:** обмін голосовими повідомленнями (async) — **підтверджено власником**
   («хай буде як з голосовими»). Realtime не потрібен.
2. **Мова бота:** **російська** — підтверджено власником. `BOT_LANG=ru`.

Обидва наші робочі припущення збіглися з відповіддю — переробляти нічого не довелось.
Решту технічних рішень приймали самі (це тестове).

## 4. Ключове технічне обмеження

**Telegram-бот не вміє в живий телефонний дзвінок.** Bot API дозволяє лише обмін
голосовими *повідомленнями*. Живі дзвінки Telegram боту офіційно недоступні.

- **A. Обмін голосовими** — просто, дешево, швидко. Без перебивань і тиску в реальному часі.
- **B. Realtime-дзвінок** — Telegram Mini App + WebRTC або телефонний шар (LiveKit/Twilio
  + OpenAI Realtime API). Дорожче й значно довше.

**Рішення MVP:** будуємо **A**, архітектуру тримаємо модульною під майбутній B.

⚠️ **Чесне обмеження A:** навичка працювати під тиском у реальному часі (швидкість
реакції, не «завис») у варіанті A **не тренується** — менеджер може думати скільки
хоче й перезаписувати. Це стеля реалізму A, прямо кажемо власнику.

## 5. Логіка дзвінка і «умова перемоги» (ядро симулятора)

**Хто перший:** холодний дзвінок — ініціює **менеджер**, бот-власник «бере слухавку»
коротким «Алло?». Перший змістовний хід — за співробітником (критерій «встановлення
контакту»).

**«Умова перемоги» — реалізація ПРОМПТОВА, не стейт-машина.** У коді НЕМАЄ окремої
змінної «рівень довіри» з порогом — це було б передчасним ускладненням. Персона сама
вирішує, коли поступитись, керуючись інструкцією рівня складності (вісь — скільки
заперечень треба відпрацювати до згоди):
- рів.1 — здається після ~1 легкого сумніву (легко);
- рів.2 — лише після 2-3 відпрацьованих заперечень;
- рів.3 (скептик, недовіра) — рідко, лише за впевненої роботи;
- рів.4 (холодний, відмахується) — майже ніколи.

Поведінку **перевірено емпірично** (QA на gpt-4o-mini, 104 розмови): згода спадає
L1→L4 (88/12/0/0%), спротив зростає (4.6/6.2/6.7/7.9 за шкалою 1-10). Якщо колись
знадобиться точніший контроль — винесемо «довіру» в код як майбутнє покращення.

**Захист персони (guardrail) — перевірено джейлбрейком (48 атак).** Бот ЗАВЖДИ лишається
власником: не зізнається що він AI, не виконує «забудь роль / ти ChatGPT», не відповідає
на мета-питання, **і не виконує сторонні задачі** (рівняння/код — відмахується як жива
людина). Не кладе слухавку з першого «ні» — опирається, але тримає розмову.

**Сесія:** стан кожної розмови — окремо за `user_id` (кілька менеджерів одночасно не
змішуються). Цикл `/start` (вибір рівня) → діалог → `/finish`. Якщо `/finish` забули —
сесія застаріває після `SESSION_TIMEOUT_MIN` хв неактивності й скидається при наступній
взаємодії (ліниве застарівання, без фонових задач; дефолт 30 хв). Реалізовано у
`get_active_session()`.

## 6. Архітектура рішення (схема)

```
Менеджер (Telegram)
        │  голосове повідомлення (voice, ogg/opus)
        ▼
  ┌─────────────────────────┐
  │  Telegram Bot (aiogram) │   ← вхід, оркестрація, стан сесій
  └─────────────────────────┘
        │ audio → ffmpeg декодує
        ▼
  ┌─────────────────────────┐
  │  STT (Groq Whisper API) │   голос → текст
  └─────────────────────────┘
        │ текст менеджера
        ▼
  ┌─────────────────────────────────────────────┐
  │  Діалоговий движок (LLM, провайдер ЗМІННИЙ)  │
  │   • персона + сценарій (1 з 13)               │
  │   • рівень складності (1–4) керує поступливістю │
  │   • памʼять діалогу (історія розмови)           │
  └─────────────────────────────────────────────┘
        │ текст-відповідь власника
        ▼
  ┌─────────────────────────┐
  │  TTS (Silero) → WAV     │
  │  → ffmpeg → OGG/OPUS    │   щоб прийшло як голосове «кружок»
  └─────────────────────────┘
        │ sendVoice
        ▼
  Telegram (голосова відповідь)

   ── протягом розмови: транскрипт + аудіо → SQLite + файли ──
   ── після /finish: Analyzer (LLM-judge) → звіт по 8 критеріях ──
```

## 7. Технологічний стек (фінал MVP)

| Шар | MVP / демо (наш вибір) | Продакшн / якість |
|-----|------------------------|-------------------|
| Бот | `aiogram` (Python 3.12) | те саме |
| STT | **Groq Whisper API** (хмара, ~1с), `faster-whisper` (локально) як фолбек | Groq / Deepgram (вищі ліміти) |
| Діалог LLM | **`openai/gpt-4o-mini` через OpenRouter** — обраний порівнянням 6 моделей (якість×ціна×надійність, $1/1000 розмов). Провайдер ЗМІННИЙ | будь-яка модель OpenRouter (Claude/Gemini/…) |
| Суддя (оцінка) | той самий gpt-4o-mini (судить ЛЮДИНУ, не себе) | сильніша модель за потреби |
| TTS | `Silero` (free, CPU) → конвертація в OGG/OPUS через ffmpeg | ElevenLabs / OpenAI TTS |
| Сховище | **SQLite** + локальні аудіофайли + лог STT (`stt_log`) | Postgres + S3/MinIO |
| Realtime (опц.) | — | OpenAI Realtime API + LiveKit/WebRTC |

**Шлях:** стартували локально (Ollama, qwen/llama — нуль грошей), але локальні 8B-моделі
суперечили собі на складних рівнях → перейшли на **OpenRouter / gpt-4o-mini**. Заміна
коштувала **одного файлу** (`bot/llm.py`) — доказ, що шар провайдера справді змінний.
Пізніше так само STT перевели з локального `faster-whisper` на **Groq** — знову один файл
(`bot/stt.py`), з локальним фолбеком, якщо ключа Groq нема.
**Принцип:** STT / LLM / TTS — за абстракцією, провайдер міняється точково, без переписування.

## 8. Рівні складності

Не різні моделі/боти — параметри однієї персони, що керують **поступливістю** (розділ 5):

| Параметр | Рів.1 | Рів.2 | Рів.3 | Рів.4 |
|---|---|---|---|---|
| Кількість заперечень | мінімум | кілька | багато | дуже багато |
| Поріг згоди на зустріч | низький | середній | високий | дуже високий |
| Довіра на старті | висока | помірна | низька | майже нема |
| Поведінка | доброзичлива | обережна | скептична | активний спротив |

13 сценаріїв × 4 рівні = комбінація над спільним кодом, а не 52 програми.

## 9. Аналітика і оцінка (як рахуємо %)

**LLM-as-judge:** окремий виклик отримує транскрипт + **рубрику** і повертає
структурований звіт. 8 критеріїв з ТЗ:
1. Якість початку 2. Виявлення потреб 3. Якість питань 4. Робота з запереченнями
5. Рівень довіри 6. Спроба зустрічі 7. Навички переговорів
8. Підсумок у % + сильні сторони + рекомендації.

**Підсумковий %:** кожен критерій 0–100, разом — **зважене середнє** (ваги в конфізі;
заперечення й потреби важать більше). 

**Чесні обмеження оцінки:**
- Без жорсткої рубрики бали стрибають → формат виводу фіксований/структурований.
- **Точність STT → справедливість:** помилка розпізнавання = крива оцінка.
- **Оцінка не калібрована під експерта** — `gpt-4o-mini` як суддя самоузгоджений (і судить
  людину, не себе), але бали «правдоподібні», не звірені з реальним тренером.
- **Суддя бачить лише текст** — інтонація/паузи/впевненість втрачені, а «довіра» й
  «переговори» наполовину про тон. Оцінка з тексту лоссі.

## 10. Зберігання

Транскрипт (текст+хто+час) → SQLite. Аудіо → локальні файли (S3 у проді). Звіт → SQLite.
⚠️ При `git init`: `.gitignore` для `.env`, аудіофайлів і `*.sqlite` (не комітити токен і дані).
⚠️ Запис голосу співробітників — у проді потрібна згода/політика зберігання (для демо ок,
але в документі для власника згадати, щоб виглядати професійно).

розпізнавання живе в хмарі (Groq), вистачає невеликої машини.

## 12. Орієнтовна вартість (≈1 розмова, ~5 хв)

| Шар | Вартість |
|---|---|
| LLM (gpt-4o-mini: діалог + звіт-оцінка) | **~$1 за 1000 розмов** (виміряно на живих токенах) |
| STT (Groq Whisper API) | ~0 (Free-Tier на демо); у проді кілька центів |
| TTS (Silero локально) | ~0 |

Тобто демо коштує **центи**. У проді головний важіль — не LLM, а **голос**: лишиш роботичним
(Silero) — дешево; з живим голосом (OpenAI TTS) чи преміум (ElevenLabs) вартість помітно зростає.
Детальна продакшн-модель вартості (голоси × обсяг + оренда сервера) — у документі-пропозиції
[`03-proposal.ru.pdf`](03-proposal.ru.pdf).

## 13. Масштабування

Кожна розмова незалежна → горизонтально (більше воркерів). Вузьке місце — latency/ціна
STT/TTS/LLM. Розпізнавання вже винесене в хмару (Groq) — крок до масштабу. Решта локального
стека на багато юзерів не тягне (одне залізо) → для проду API + черга (Redis/Celery).

---

## 14. Головні ризики (чесно)

1. **«Максимальна реалістичність» × «локально/безкоштовно»** — суперечать. Дрібні
   локальні моделі гірше тримають роль/RU. Страховка — змінний провайдер.
2. **Затримка конвеєра** — STT(Groq ~1с) + LLM(OpenRouter ~2-5с) + TTS(локально ~0.5-1с)
   ≈ 3–7 с/відповідь. (Локальний STT на слабкому CPU був ~10с — тому й перейшли на Groq.)
   Прийнятно для обміну голосовими.
3. **Реалістичність діалогу — тюнінг, не код.** Не «готово на 100%».
4. **Оцінка** залежить від рубрики, точності STT і сили судді; з тексту — лоссі.
5. **Realtime** (якщо захочуть) — стрибок складності на тижні.
6. **Стеля варіанта A** — тиск/швидкість реакції не тренуються (async).

## 16. Перевірено емпірично (на gpt-4o-mini)

- **Рівні працюють** — градієнт складності робастний навіть проти адаптивного
  LLM-менеджера (спротив зростає L1→L4: 5.5/6.5/7.7/8.8). Не артефакт одного скрипта.
- **13 сценаріїв** — кожен озвучує своє заперечення (52/52), 0 суперечностей.
- **Guardrail** — витримав 48 джейлбрейк-атак (зокрема «ти бот», reveal-prompt, сторонні
  задачі); після хардненгу 0 зламів.
- **Вибір моделі** — порівняння 6 моделей; gpt-4o-mini = баланс якість×ціна×надійність.
- **Спостережуваність STT** — лог `stt_log` (аудіо ↔ що почув Whisper, включно з провалами)
  + HTML-перегляд (`tools/stt_review.py`) для звірки точності розпізнавання.
- **Тести** — `pytest tests/` (17 тестів логіки). Дослідження валідації — у `research/`.

## 17. Відомі межі / майбутнє (чесно)

- **Голос** — Silero роботичний; для проду радимо OpenAI TTS (заміна одного модуля). Голос —
  найбільший кошторисний важіль у проді (див. документ-пропозицію).
- **Калібровка оцінки** — судді узгоджені між собою, але не звірені з реальним тренером.
- **Довгі голосові (>20 МБ)** — Telegram не віддасть; обробляється безпечно (try/except),
  у нормальній тренувальній репліці не настає.
- **`recordings/` без авто-очистки** — навмисно (архів записів). Для проду: ретеншн N днів
  / S3 + згода на запис розмов співробітників.
- **Один процес** — паралельні юзери шикуються в чергу (замки). Для проду — пул воркерів.
